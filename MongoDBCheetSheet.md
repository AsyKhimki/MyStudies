<h4> 3. Indexes </h4> 

| Nr             | Topic| Notes &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp;  | Reference |
| :------| :------ |:---- | :----------------|
3.1  | Given a query that is performing a collection scan, identify which index would improve the performance of this query. |- there is one devault index per collection: `_id` </br> - Follow this order: Equality, Sort, Range </br></br> **General rules** </br> - **Order matters**: MongoDB can only use the index if the query references the fields in the order they appear in the compound index `{"product": 1, "price": 1}` might improve the performance whereas `price: 1, product: 1` will not </br> - **Leftmost prefix rule**: To use a compound index, the query must reference the leftmost field of the index. If the leftmost field is not included in the query, MongoDB cannot use the index. </br> - **Sort direction:** MongoDB can only use an index for sorting if the sort direction matches the direction specified in the index. |
3.2  | Given a query that is performing a collection scan on an equality match on an array field, identify which index would improve the performance of this query. |`db.col.createIndex({fieldname: 1})`</br> `db.col.createIndex({fieldname1: 1}, {unique: true})`|
3.3  | Given a query with no constraint and a sort of two fields that is doing collection scan, identify which index would improve the performance of this query. | **Most common index types** </br> - single field </br> - compound </br> - multikey indexes operate on an array field </br></br> **Important** </br>- Compound indexes are optimized to be used when the query matches the fields in a specific order (starting from the leftmost field) |
3.4  | Given a collection, identify how many indexes exist for that collection.|`db.coll.getIndexes()`|
3.5  | Identify the trade-offs of using indexes and the ramifications of deleting indexes support queries |`db.coll.hideIndex('active_1_birthdate_-1_name_1')` </br>`db.coll.hideIndex({active: 1, birthdate: -1, name:1 })`</br>`db.coll.dropIndex('active_1_birthdate_-1_name_1')`</br> </br> **Improve performance**</br> - speed up queries </br> - reduce disk I/O </br> - Reduce resources required </br> - Support equality matches and range-based operations and return sorted results </br> </br>**Without Index** </br> - MongoDB reads all documents (collection scan) </br> - Sorts results in memory </br> </br> **Problems with indexes** </br> - insert or update document: need to update index </br> - Indexes have a write cost.|
3.6  | Identify the explain plan outputs that signify a potential performance issue, specifically whether an index is present or not for the given query.|`db.collection.find(<query>).explain()`</br> </br> **Important Fields to Look For**</br> - **COLLSCAN** (Collection Scan): MongoDB scanned the entire collection. This generally indicates poor performance because itâs not using an index. </br> - **IXSCAN** (Index Scan): MongoDB used an index to fetch the documents, which is much more efficient for performance. </br> -`totalDocsExamined` much higher than `nReturned`: could indicate that query is inefficient or that need an index on the field being queried </br> - Index-Only Queries: If all the fields you need are covered by the index, MongoDB can avoid the FETCH stage entirely.
